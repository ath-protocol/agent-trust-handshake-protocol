本示例演示了一个完整的 ATH 流程：旅行代理代表用户检查航班确认邮件。

## 参与方

- **用户**：Alice，拥有一个电子邮件账户
- **代理**：TravelBot（`https://travel-agent.example.com/.well-known/agent.json`）
- **服务**：一个电子邮件服务
- **ATH 实现者**：网关或原生服务

## 阶段 A：代理注册（一次性操作）

1. TravelBot 开发者向 ATH 实现者注册：
   - 提交代理身份、证明 JWT，并请求 `mail:read` 权限范围

2. 实现者审核并批准 TravelBot 使用 `mail:read`

3. TravelBot 收到 `client_id`

## 阶段 B：运行时（Alice 想要查看邮件）

**步骤 1**：Alice 告诉 TravelBot："帮我查找东京旅行的确认邮件"

**步骤 2**：TravelBot 发送授权请求：

```
POST /ath/authorize
{
  "client_id": "ath_travelbot_001",
  "provider_id": "example-mail",
  "scopes": ["mail:read"]
}
```

**步骤 3**：实现者验证：
- TravelBot 已注册且已批准（**应用侧授权** ✓）
- TravelBot 已获批 `mail:read` ✓

**步骤 4**：实现者发起 OAuth 流程 → 跳转到电子邮件服务同意页面

**步骤 5**：Alice 看到同意页面：

> "TravelBot 想要查看你的电子邮件"

**步骤 6**：Alice 批准（**用户侧授权** ✓）

**步骤 7**：实现者收到 OAuth 令牌，计算权限范围交集：

```
代理已批准：  mail:read  ✓
用户已同意：  mail:read  ✓
有效权限范围：mail:read
```

**步骤 8**：实现者向 TravelBot 签发 ATH 访问令牌，绑定到：
- 代理：TravelBot
- 用户：Alice
- 提供者：example-mail
- 权限范围：mail:read

**步骤 9**：TravelBot 调用邮件 API：

```
GET /ath/proxy/example-mail/v1/messages?q=Tokyo+trip
Authorization: Bearer ath_tk_xxxxxxxx
```

**步骤 10**：实现者验证令牌，处理/代理转发请求，返回结果

**步骤 11**：TravelBot 为 Alice 汇总确认邮件

## 如果 TravelBot 尝试发送邮件会怎样？

如果 TravelBot 尝试调用发送端点：

```
POST /ath/proxy/example-mail/v1/messages/send
Authorization: Bearer ath_tk_xxxxxxxx
```

实现者会**拒绝**该请求，因为：
- ATH 令牌仅绑定了 `mail:read`
- `mail:send` 从未在有效权限范围内
- 可信握手确保代理不能超出其已批准的能力范围
