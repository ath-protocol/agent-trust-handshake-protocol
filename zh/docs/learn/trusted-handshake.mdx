可信握手是 ATH 协议的核心安全原则。它要求**服务方和用户双方**都必须批准，代理才能访问任何资源。

## 核心原则

只有**两个条件同时满足**时，代理才能访问服务：

| 授权类型 | 谁来决定 | 回答的问题 |
|---|---|---|
| **应用侧** | 服务提供商 / 代理注册表 | "该代理是否被允许使用此服务？" |
| **用户侧** | 终端用户 | "该用户是否允许此代理代表自己行动？" |

任何单一授权都不够：
- 如果应用未批准该代理，用户的同意无关紧要
- 如果用户未同意，应用对该代理的批准无关紧要

## 两阶段流程

可信握手由两个不同的阶段组成：

### 阶段 A：应用侧授权（一次性设置）

代理开发者向 ATH 实现方注册代理。这是一个一次性过程，用于建立代理的身份和批准的能力。

```
Agent Developer                    ATH Implementor
      │                            （网关或服务）
      │                                    │
      │  1. 注册代理                       │
      │  （agent_id、证明、               │
      │   请求的能力）                     │
      ├───────────────────────────────────►│
      │                                    │
      │                          2. 验证代理身份
      │                             审核能力
      │                             批准/拒绝
      │                                    │
      │  3. 注册响应                       │
      │  （client_id、已批准的范围、      │
      │   批准有效期）                     │
      │◄───────────────────────────────────┤
```

**结果**：代理以特定的能力限制被纳入已批准的注册表中。此状态持续有效直到被撤销。

### 阶段 B：用户侧授权（按用户、运行时）

当用户希望代理代表自己访问服务时，会发起标准的 OAuth 同意流程——但前提是代理已通过阶段 A。

```
User         Agent        ATH Implementor      Service
 │             │                  │                │
 │  "执行 X"  │                  │                │
 ├────────────►│                  │                │
 │             │  请求访问        │                │
 │             ├─────────────────►│                │
 │             │        验证代理身份               │
 │             │        检查批准状态               │
 │             │        发起 OAuth 流程            │
 │             │                  ├───────────────►│
 │  OAuth 同意页面               │                │
 │◄──────────────────────────────────────────────┤
 │  用户批准                     │                │
 ├──────────────────────────────────────────────►│
 │             │                  │  OAuth 令牌    │
 │             │                  │◄───────────────┤
 │             │       范围交集                    │
 │             │       与令牌绑定                  │
 │             │  ATH 访问令牌                     │
 │             │◄─────────────────┤                │
 │             │  API 调用        │                │
 │             ├─────────────────►│  代理/服务     │
 │             │                  ├───────────────►│
 │             │                  │◄───────────────┤
 │             │◄─────────────────┤                │
 │◄────────────┤                  │                │
```

<Note>
在网关模式下，"ATH 实现方"是网关，"Service"是上游提供商。在原生模式下，"ATH 实现方"和"Service"是同一个实体。
</Note>

## 为什么两个阶段都很重要

没有阶段 A（应用侧授权）：
- 任何代理都可以向任何服务发起 OAuth 流程
- 恶意代理可以欺骗用户授予访问权限
- 服务运营商无法了解哪些代理在访问其 API

没有阶段 B（用户侧授权）：
- 代理可以在用户不知情的情况下访问用户数据
- 没有针对数据访问的按用户同意机制
- 违反了已建立的 OAuth 隐私预期
