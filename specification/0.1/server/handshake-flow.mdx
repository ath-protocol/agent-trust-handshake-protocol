# ATH Protocol Server Handshake Flow
This document describes the handshake process from the server (receiving agent) perspective.
## Server Role
The server is the party that receives the handshake request, authenticates the client, and negotiates permissions.
## Complete Server-side Flow
| Step | Operation | Description |
|------|-----------|-------------|
| 1 | Receive Handshake Request | Parse client's handshake request, extract DID and capability information |
| 2 | Verify Client Identity | Send client's DID to Identity Registry for validity verification |
| 3 | Process Verification Result | If client identity is invalid, return error and terminate handshake |
| 4 | Send Handshake Response | Return server's own DID, capability list, and supported algorithms |
| 5 | Receive Permission Request | Parse client's permission request list, evaluate permission requirements |
| 6 | Approve/Deny Permissions | Based on policy, return permission approval result (approve/deny/partial) |
| 7 | Receive Permission Confirmation | Confirm the final permission scope with the client |
| 8 | Process Key Negotiation Request | Parse client's key negotiation request, select appropriate encryption algorithm |
| 9 | Send Key Negotiation Response | Return key negotiation response, complete key exchange |
| 10 | Establish Encrypted Session | Generate shared session key, establish end-to-end encrypted communication channel |
## Server Responsibilities
1. Strictly verify the identity of all requesting clients
2. Implement permission control policies based on least privilege principle
3. Ensure the security of session key generation and storage
4. Enforce permission limits during business processing
5. Maintain complete audit logs of all handshake and access processes
