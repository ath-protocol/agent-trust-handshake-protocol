# Server-Side Handshake Flow
This document describes the handshake process from the server's perspective, following the standard 8-step ATH protocol flow.
## 8-Step Flow for Server Implementation
### Step 1: Process Client Identity Announcement
**Server Responsibility**: Validate and process client's identity announcement
- Record client DID and capabilities
- Verify any provided client identity credentials
- Determine highest mutually supported protocol version
### Step 2: Send Identity Response
**Server Responsibility**: Respond with server identity information
- Include server DID identifier
- Announce negotiated protocol version
- Provide server capabilities and metadata (endpoints, supported scopes, token formats, etc.)
- Optionally attach server identity credentials
### Step 3: Process Scope Request
**Server Responsibility**: Receive and validate client's scope request
- Parse requested permissions and validity period
- Validate request against server security policies
- Check client's permission eligibility
### Step 4: Send Scope Negotiation Response
**Server Responsibility**: Perform scope negotiation based on security policies
- Approve or reject requested scopes based on access control policies
- Provide clear reasons for rejected scopes
- Attach any access restrictions (IP limits, rate limits, etc.) to approved scopes
### Step 5: Perform Identity Verification
**Server Responsibility**: Verify client identity authenticity
- Generate and send random challenge string to client
- Verify client's signature using client's public key
- Ensure client possesses the private key corresponding to the claimed DID
- Optionally respond to client challenge for mutual authentication
### Step 6: Issue Access Credential
**Server Responsibility**: Generate and issue signed access token
- Create token containing agreed scopes, validity period, and party identifiers
- Sign token using server's private key
- Support multiple token formats (JWT, PASETO, etc.)
### Step 7: Process Credential Confirmation
**Server Responsibility**: Validate client's credential confirmation
- Verify client has successfully received and validated the token
- Record active token for future access control
- Prepare for business request processing
### Step 8: Complete Handshake
**Server Responsibility**: Finalize handshake process
- Send final confirmation message to client
- Establish trusted communication session
- Enforce scope-based access control for all subsequent requests
## Server Implementation Requirements
- Implement fine-grained access control policies for scope negotiation
- Use secure random number generation for challenges
- Support token revocation mechanism
- Implement comprehensive audit logging for all handshake events


## Authorization Policy
### Scope Validation
Servers must validate scope requests against:
- Client identity and trust level
- Resource owner consent (if applicable)
- Organizational access control policies
- Rate limiting rules
### Policy Decision
Authorization decisions can be:
- ✅ **Approve**: Full scope request approved
- ⚠️ **Partial**: Subset of scopes approved
- ❌ **Deny**: No scopes approved, handshake terminated


## Server Registration & Identity
### Server DID Format
Server DID follows the format: `did:ath:server:<domain>`
Example: `did:ath:server:api.example.com`
### Well-Known Configuration
Servers must expose a well-known configuration endpoint at `/.well-known/ath-protocol.json`:
```json
{
  "did": "did:ath:server:api.example.com",
  "public_key": "-----BEGIN PUBLIC KEY-----
...",
  "versions": ["0.1"],
  "scopes_supported": ["user:read", "data:write", "payment:initiate"],
  "token_formats": ["jwt", "paseto"]
}
```


## Token Specification
### Token Formats
Supported token formats:
1. **JWT (JSON Web Token)** - RFC 7519 compliant
2. **PASETO (Platform-Agnostic Security Tokens)** - RECOMMENDED
### Token Claims
Standard claims included in all tokens:
- `iss`: Server DID
- `sub`: Client DID
- `scope`: Space-separated list of approved scopes
- `iat`: Issued timestamp
- `exp`: Expiration timestamp
- `jti`: Unique token ID
### Token Signing
Tokens must be signed using:
- Ed25519 (RECOMMENDED)
- secp256k1
- HS256/HS512 (only for symmetric use cases)


## Token Revocation
### Revocation Endpoint
Servers must provide a revocation endpoint: `POST /ath/v1/revoke`
### Revocation Reasons
Tokens can be revoked for:
- Client requested logout
- Security compromise detected
- Scope permissions changed
- Token expiration (automatic)
### Revocation List
Servers should maintain a token revocation list (TRL) or use short-lived tokens to minimize revocation overhead.


## Proxy Mode Support
Servers can operate as ATH protocol proxies:
- Forward requests to backend services
- Validate tokens before forwarding
- Add additional security headers and logging
- Implement cross-domain access control
### Proxy Headers
When proxying, servers add the following headers:
- `X-ATH-Client-DID`: Authenticated client DID
- `X-ATH-Scopes`: Approved scopes for the request
- `X-ATH-Token-ID`: Unique token ID for audit logging
