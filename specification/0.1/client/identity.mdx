## Agent Identity (Agent_ID)

Each agent MUST have a unique and verifiable identity.

**Format**: A URI-based identifier following this pattern:

```
https://<agent-domain>/.well-known/agent.json
```

**Examples**:
```
https://travel-agent.example.com/.well-known/agent.json
https://coding-assistant.example.com/.well-known/agent.json
```

## Agent Identity Document

An agent MUST publish a JSON document at its `agent_id` URI:

```json
{
  "ath_version": "0.1",
  "agent_id": "https://travel-agent.example.com/.well-known/agent.json",
  "name": "TravelBot",
  "developer": {
    "name": "Example Corp",
    "id": "dev-example-12345",
    "contact": "security@example.com"
  },
  "capabilities": ["flight-search", "hotel-booking", "itinerary-planning"],
  "public_key": "<JWK or PEM for agent attestation verification>"
}
```

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `ath_version` | string | Protocol version (e.g., `"0.1"`) |
| `agent_id` | string | Canonical URI of this agent |
| `name` | string | Human-readable agent name |
| `developer` | object | Developer information |
| `developer.name` | string | Developer or organization name |
| `developer.id` | string | Developer identifier |
| `public_key` | string/object | JWK or PEM for attestation verification |

### Optional Fields

| Field | Type | Description |
|-------|------|-------------|
| `developer.contact` | string | Security contact email |
| `capabilities` | string[] | List of agent capabilities |

## Service Identity (App_ID)

Services are identified by their existing identifiers:
- **Client applications**: Package name (e.g., `com.example.mail`)
- **Server-side services**: URI (e.g., `https://api.example.com`)

## Agent Attestation

Agents use signed JWTs (agent attestation tokens) to prove their identity.

### JWT Header

```json
{
  "alg": "ES256",
  "typ": "JWT",
  "kid": "<key-id>"
}
```

### JWT Payload

```json
{
  "iss": "https://travel-agent.example.com",
  "sub": "<agent_id>",
  "aud": "<target-service-or-gateway>",
  "iat": 1720000000,
  "exp": 1720003600,
  "jti": "<unique-token-id>",
  "capabilities": ["flight-search"]
}
```

### Required Claims

| Claim | Description |
|-------|-------------|
| `iss` | Issuer — the agent's domain |
| `sub` | Subject — the full `agent_id` URI |
| `aud` | Audience — the target service or gateway URL |
| `iat` | Issued At — timestamp. Implementers MUST verify that `iat` is within 5 minutes of the current time |
| `exp` | Expiration Time — MUST be present |
| `jti` | JWT ID — MUST be unique per attestation. Implementers MUST maintain a replay cache of seen `jti` values (retained until `exp`) and reject duplicates |

### Verification

The verifier (gateway or service):
1. Fetches the agent's identity document from the `agent_id` URI
2. Extracts the public key
3. Verifies the JWT signature using the public key
4. Validates `exp` (not expired) and `aud` (matches the verifier's URL)
5. Verifies that `jti` has not been seen before (checks against replay cache)
6. Verifies that `iat` is within 5 minutes of the current time
7. Implementers MUST reject attestations that are expired, have mismatched audiences, or have duplicate `jti` values

### Identity Document Caching

- Implementers MAY cache agent identity documents to reduce fetch overhead
- The maximum cache time-to-live MUST NOT exceed **5 minutes**
- Agent identity documents SHOULD include a `Cache-Control: max-age=300` response header
- When attestation verification fails due to a signature mismatch (not expiry or audience errors), implementers SHOULD re-fetch the identity document (bypassing cache) to support emergency key rotation. To prevent abuse as a DoS amplifier, implementers MUST rate-limit re-fetches to at most once per `agent_id` per minute
- Implementers SHOULD set connect and read timeouts when fetching agent identity documents
