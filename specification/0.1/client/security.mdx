## Agent Attestation Verification

- Agent attestation JWTs MUST be verified using the public key published by the agent at its `agent_id` URI
- JWTs MUST include an expiration time (`exp`) and audience (`aud`)
- Implementers MUST reject attestations that are expired or have mismatched audiences

## Token Binding

- ATH access tokens MUST be bound to a specific `(agent_id, user_id, provider_id, scopes)` tuple
- A token obtained by one agent MUST NOT be used by another agent
- A token obtained for one user MUST NOT be used by another user

## OAuth Security

- All OAuth authorization requests MUST use PKCE ([RFC 7636](https://datatracker.ietf.org/doc/html/rfc7636)) with the S256 challenge method
- ATH token exchange requests (`POST /ath/token`) use the `application/json` content type (as defined by the JSON Schema and OpenAPI). When the ATH implementer exchanges the authorization code with the upstream OAuth authorization server on behalf of the agent, that upstream request MUST use `application/x-www-form-urlencoded` in accordance with [RFC 6749 §4.1.3](https://datatracker.ietf.org/doc/html/rfc6749#section-4.1.3)
- Service provider OAuth client secrets MUST be stored securely and never exposed to agents or users
- Service provider tokens (access_token, refresh_token) obtained from the OAuth flow MUST be stored server-side and never sent to agents

## Transport Security

- All ATH communications MUST use HTTPS (TLS 1.2+)
- Implementers SHOULD verify TLS certificates of upstream services
- In high-security environments, mTLS is OPTIONAL (v2 consideration)

## Audit Logging

Implementers SHOULD log all authorization decisions:

```json
{
  "timestamp": "2026-04-10T12:00:00Z",
  "event": "access_granted",
  "agent_id": "https://travel-agent.example.com/.well-known/agent.json",
  "user_id": "user-12345",
  "provider_id": "example-mail",
  "requested_scopes": ["mail:read", "mail:send"],
  "effective_scopes": ["mail:read"],
  "denied_scopes": ["mail:send"],
  "denial_reason": "agent not approved for mail:send"
}
```

## Rate Limiting and Abuse Prevention

- Implementers SHOULD enforce per-agent rate limits
- Agents with abnormal request patterns SHOULD be flagged and potentially suspended
- Agent registries SHOULD support revocation for compromised agents
