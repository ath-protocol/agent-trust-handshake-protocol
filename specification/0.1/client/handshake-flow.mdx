# ATH Protocol Client Handshake Flow
This document describes the handshake process from the client (initiating agent) perspective.
## Client Role
The client is the party that initiates the handshake request, actively establishing trust connection with the server.
## Complete Client-side Flow
| Step | Operation | Description |
|------|-----------|-------------|
| 1 | Send Handshake Request | Construct and send handshake request containing client DID, capability list, and supported algorithms |
| 2 | Wait for Server Response | Wait for server's handshake response containing server DID and capabilities |
| 3 | Verify Server Identity | Send server's DID to Identity Registry for verification |
| 4 | Process Verification Result | If server identity is invalid, terminate the handshake process |
| 5 | Send Permission Request | Construct permission request list based on business requirements |
| 6 | Process Permission Response | Review server's permission approval result, check if it meets business needs |
| 7 | Send Permission Confirmation | Confirm the agreed permission scope with the server |
| 8 | Initiate Key Negotiation | Send session key negotiation request, supported encryption algorithms |
| 9 | Process Key Response | Verify server's key negotiation response, generate shared session key |
| 10 | Establish Encrypted Session | Complete key negotiation, establish end-to-end encrypted communication channel |
## Client Responsibilities
1. Ensure the authenticity and validity of its own identity information
2. Request only the minimum necessary permissions required for business
3. Properly store and use the negotiated session key
4. Comply with the permission scope agreed during negotiation
5. Record all handshake processes for audit purposes
