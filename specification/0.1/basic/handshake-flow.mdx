# ATH Protocol Handshake Flow Specification
## Overview
The ATH protocol handshake process consists of 3 core phases and 12 standard steps, enabling secure trust establishment between heterogeneous AI agents.
## 3 Core Phases
### Phase 1: Identity Verification (6 steps)
Mutual identity verification through decentralized identity registry, ensuring both parties are legitimate.
### Phase 2: Permission Negotiation (3 steps)
Fine-grained permission negotiation based on OAuth 2.0 extension framework, implementing least privilege principle.
### Phase 3: Session Establishment (3 steps)
Establish end-to-end encrypted communication session using TLS 1.3 protocol.
## Complete 12-Step Handshake Flow
| Step | Phase | Description |
|------|-------|-------------|
| 1 | Identity Verification | Agent A sends handshake request to Agent B, including DID (Decentralized Identifier) and capability list |
| 2 | Identity Verification | Agent B sends Agent A's identity information to Identity Registry for validity verification |
| 3 | Identity Verification | Identity Registry returns identity verification result to Agent B |
| 4 | Identity Verification | Agent B returns handshake response to Agent A, including its own DID and capability list |
| 5 | Identity Verification | Agent A sends Agent B's identity information to Identity Registry for validity verification |
| 6 | Identity Verification | Identity Registry returns identity verification result to Agent A |
| 7 | Permission Negotiation | Agent A sends permission request list to Agent B, including requested interfaces and data scope |
| 8 | Permission Negotiation | Agent B returns permission approval result to Agent A (approve/deny/partial approve), including validity period |
| 9 | Permission Negotiation | Agent A sends permission scope confirmation to Agent B |
| 10 | Session Establishment | Agent A sends session key negotiation request to Agent B |
| 11 | Session Establishment | Agent B returns session key negotiation response to Agent A |
| 12 | Session Establishment | Both parties establish encrypted communication session |
## Key Features
1. **Decentralized Identity**: Uses DID as unique identifier, no single point of failure
2. **Mutual Verification**: Both parties verify each other's identity through independent registry
3. **Fine-grained Permission**: Supports interface-level, data-level and time-level permission control
4. **End-to-end Encryption**: TLS 1.3 based encryption, ensuring communication security
5. **Auditable**: All handshake processes are recorded for audit and traceability
