This example walks through a complete ATH flow where a travel agent checks flight confirmation emails on behalf of a user.

## Participants

- **User**: Alice, has an email account
- **Agent**: TravelBot (`https://travel-agent.example.com/.well-known/agent.json`)
- **Service**: An email service
- **ATH Implementor**: Gateway or native service

## Phase A: Agent Registration (One-Time)

1. TravelBot developer registers with the ATH implementor:
   - Submits agent identity, attestation JWT, and requests `mail:read` scope

2. The implementor reviews and approves TravelBot for `mail:read`

3. TravelBot receives `client_id`

## Phase B: Runtime (Alice Wants to Check Emails)

**8-Step Handshake Flow:
1. Client Identity Announcement
2. Server Identity Response
3. Scope Request
4. Scope Negotiation
5. Identity Verification
6. Credential Issuance
7. Credential Confirmation
8. Handshake Complete

## What If TravelBot Tries to Send Email?

If TravelBot attempts to call a send endpoint:

```
POST /ath/proxy/example-mail/v1/messages/send
Authorization: Bearer ath_tk_xxxxxxxx
```

The implementor **rejects** the request because:
- The ATH token is bound to `mail:read` only
- `mail:send` was never in the effective scope
- The trusted handshake ensures the agent cannot exceed its approved capabilities
